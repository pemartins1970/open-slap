## B-01 — Diagnóstico Original vs Realidade

### Diagnóstico Original (Backlog)
> Experts sem BaseAgent caem no fallback do orchestrator. O fallback chama
> llm_manager mas o expert é passado como string em vez de dict → `'str' object
> has no attribute 'get'`.

### O Código Real diz o contrário

O MoE Router (`backend/moe_router_simple.py:621-678`) retorna `expert` como
`Dict[str, Any]` em TODOS os cenários. O fallback do orchestrator
(`backend/ws/orchestrator.py:181-201`) passa esse dict corretamente para
`llm_manager.stream_generate()`. Este caminho **funciona**.

O problema real está em outro lugar:

### Causa Raiz Real — Duas Camadas

#### Bug A — `context={}` é falsy em Python (CRÍTICO)

**Arquivo:** `backend/routes/feedback_plan_routes.py:66`

```python
agent = agent_registry.get(skill_id) if skill_id else None
if agent:
    result = await agent.execute(title, context={})  # <-- LINHA 66
```

Quando um plano é aprovado e executado via `_run_orchestration`, o `context={}`
(dict vazio) é passado para `BaseAgent.execute()`.

**Arquivo:** `backend/agents/base.py:21`

```python
expert_name = context.get("expert", self.name) if context else self.name
```

Em Python, `bool({})` é `False`. Portanto, a expressão cai no `else`:
`self.name` — uma **string** (ex: `"backend"`, `"frontend"`).

**Arquivo:** `backend/agents/base.py:26`

```python
async for chunk in llm_manager.stream_generate(
    prompt=prompt, expert=expert_name, ...  # expert_name é str!
):
```

**Arquivo:** `backend/llm_manager_simple.py:377**

```python
expert_prompt = (expert or {}).get("prompt") or ""
```

`"backend".get("prompt")` → **`AttributeError: 'str' object has no attribute
'get'`**.

**Resultado:** O erro é capturado em `base.py:36` e retornado como
`AgentResult(status="error")`. A task é marcada como `failed` no banco. **Nada
aparece no chat.**

#### Bug B — `_run_orchestration` sem WebSocket

**Arquivo:** `backend/routes/feedback_plan_routes.py:39`

A função `_run_orchestration` não recebe um parâmetro `websocket`. Resultados
são salvos no banco via `save_message()`, mas o cliente WebSocket nunca recebe
notificação em tempo real. O frontend precisa recarregar manualmente para ver
resultados.

Compare com a versão em `backend/main_auth.py:5137` que TEM o parâmetro
`websocket: Optional[WebSocket] = None`.

### Paradoxo

O caminho que o backlog chama de "fallback" (else na linha 79 de
`feedback_plan_routes.py`) usa `moe_router.select_expert()` que retorna um dict
válido — **funciona corretamente**. O caminho "normal" (agente encontrado no
registry) **está quebrado** porque passa `context={}`.

| Cenário | Encontrou agent? | Resultado |
|---------|-----------------|-----------|
| Expert com BaseAgent mapeado | Sim (`agent_registry.get()` ok) | **QUEBRADO** — `context={}` causa TypeError |
| Expert sem BaseAgent | Não (fallback) | **FUNCIONA** — `moe_router.select_expert()` retorna dict |

## B-02 — Diagnóstico Original vs Realidade

### Diagnóstico Original (Backlog)
> Sistema usa a primeira chave cadastrada até esgotar quota, depois falha.
> `_load_providers()` carrega apenas a primeira chave da lista.

### O Código Real — Três Problemas Distintos

#### Problema 1 — Nenhuma rotação de chave DENTRO de uma requisição

`stream_generate()` tem um loop de fallback ENTRE providers (linha 651), mas
NÃO tem um loop DENTRO do mesmo provider para tentar a próxima chave.

**Arquivo:** `backend/llm_manager_simple.py:180-196`

```python
def _get_next_key(self, provider_id: str) -> Optional[str]:
    provider = self.providers.get(provider_id)
    if not provider or "keys" not in provider:
        return None                                          # ← OpenAI/OpenRouter sempre retornam None
    current_index = provider["current_key_index"]
    key = _sanitize_api_key(keys[current_index])
    provider["current_key_index"] = (current_index + 1) % len(keys)  # round-robin
    return key or None
```

`_get_next_key()` implementa round-robin **entre requisições** (cada chamada a
`_stream_gemini` avança o índice), mas é chamada UMA ÚNICA VEZ por requisição.
Se a chave retornar 429, a exceção é lançada imediatamente e o controle passa
para o PRÓXIMO provider no loop de fallback — sem tentar a próxima chave do
mesmo provider.

#### Problema 2 — Estrutura de dados inconsistente entre providers

Gemini e Groq usam `"keys": [...]` (lista, suporta múltiplas chaves). OpenAI e
OpenRouter usam `"key": "sk-..."` (string única). `_get_next_key()` verifica
`"keys" not in provider` e retorna `None` para os dois últimos.

| Provider | Campo de chave | Suporta múltiplas? | Rotação funciona? |
|----------|---------------|-------------------|-------------------|
| Gemini   | `keys: list`  | Sim (env: `GEMINI_API_KEYS` CSV) | Entre requests, não dentro |
| Groq     | `keys: list`  | Sim (env: `GROQ_API_KEYS` CSV) | Entre requests, não dentro |
| OpenAI   | `key: str`    | Não | N/A |
| OpenRouter | `key: str`  | Não | N/A |

#### Problema 3 — Banco de dados de chaves ignorado pelo runtime

O sistema possui:
- Tabela `user_llm_provider_keys` em `db.py` com CRUD completo
- Endpoints REST `/api/settings/llm/keys` (GET, POST, DELETE)
- Endpoint `/api/settings/llm/keys/{provider}/active` (selecionar chave ativa)
- Função `loadProviderKeys()` em `useLLMConfig.js`

Mas `_load_providers()` no `llm_manager_simple.py:79-173` **só lê variáveis de
ambiente**. Chaves cadastradas via UI/settings nunca são carregadas pelo
runtime. A função `_get_user_api_key()` no `orchestrator.py:65` também não
integra com esta tabela.

### Fluxo de Falha (429)

1. Usuário envia mensagem → MoE seleciona expert → orchestrator chama
   `llm_manager.stream_generate()`
2. `stream_generate()` itera `self.provider_order` (gemini → groq → ollama → openai → openrouter)
3. Para Gemini: `_get_next_key("gemini")` → retorna primeira chave
4. Gemini responde 429 (quota exhausted)
5. `_stream_gemini` levanta exceção (linha 861)
6. `stream_generate()` captura no `except` (linha 739), loga, faz `continue`
7. Próximo provider: Groq → mesmo ciclo
8. Todos os providers falham → `"Todos os providers indisponíveis"`

A chave alternativa de Gemini NUNCA é tentada — o sistema pula direto para Groq.

## Impacto e Urgência

### B-01 — Impacto ALTO (crítico)
- Qualquer plano com task cujo `skill_id` corresponda a um agente registrado
  (`backend`, `frontend`, `devops`, `cto`, `general`) **nunca produz output**.
- O erro é silencioso: task aparece como "failed" no banco, sem mensagem no
  chat. Usuário vê o plano aprovado e "execução concluída", mas sem resposta.
- Skills mapeadas para agentes: backend, frontend, devops, cto, general
- Skills que CAEM no fallback (funcionam): cfo, coo, software_operator,
  ide_editor, security, data, project, pmo, po, documentation

### B-02 — Impacto ALTO (produtivo)
- Usuários com múltiplas chaves free-tier não têm fallback automático
- Uma única chave exaurida por provider quebra aquele provider inteiro
- Chaves cadastradas via UI são ignoradas completamente pelo runtime
- Produto promete (via UI de settings) uma funcionalidade que não existe

## Correções Propostas

### Fix B-01 (2 arquivos, ~5 linhas)

**1. `backend/agents/base.py:21`** — Tratar dict vazio como contexto válido:

```python
# ANTES
expert_name = context.get("expert", self.name) if context else self.name

# DEPOIS
expert_name = context.get("expert", self.name) if isinstance(context, dict) else self.name
```

**2. `backend/routes/feedback_plan_routes.py:66`** — Passar contexto com
expert e combined_context:

```python
# ANTES
result = await agent.execute(title, context={})

# DEPOIS
result = await agent.execute(title, context={"expert": skill_id, "combined_context": ""})
```

### Fix B-02 (4 áreas)

**1. Loop de retry com próxima chave ANTES de mudar de provider**

Em `stream_generate()` (linha 651), antes de `continue` para o próximo
provider, implementar loop interno: esgotar todas as chaves do provider atual
antes de tentar outro.

**2. Unificar estrutura de chaves**

Alterar `_load_providers()` para usar formato `"keys": [...]` para TODOS os
providers (incluindo OpenAI e OpenRouter), com fallback para `OPENAI_API_KEY`
como lista de 1 elemento.

**3. Integrar banco de dados de chaves**

Fazer `_load_providers()` (ou uma nova `_load_user_providers()`) consultar a
tabela `user_llm_provider_keys` e mesclar com as chaves de ambiente. Usuário
logado usa chaves do DB; usuário anônimo usa chaves de ambiente.

**4. UI de múltiplas chaves (se for manter a feature)**

Ativar `loadProviderKeys()` no `LLMConfigPanel.jsx` e renderizar gestão de
múltiplas chaves por provider. Caso contrário, remover a promessa da interface.
