## SPEC B-06 — Contexto de conversa não persiste entre mensagens

**Prioridade:** CRÍTICA
**Tipo:** Bug
**Impacto:** Sabrina trata cada mensagem como a primeira da conversa. Memória de curto prazo = zero. Invalida o valor percebido de toda interação com mais de 1 turno.

---

### Descrição do problema

O `stream_generate` (ou equivalente em `ws/orchestrator.py`) monta o payload para o LLM com apenas a mensagem corrente. O histórico da conversa ativa no DB não é consultado antes da chamada. Resultado: o modelo não tem contexto de turnos anteriores.

---

### Root cause — verificar antes de implementar

Antes de escrever qualquer linha, confirmar no código:

1. Onde exatamente o array `messages` / `contents` é montado para a chamada ao LLM.
   *Suspeito: função em `ws/orchestrator.py` ou no adapter de provider.*
2. Qual é o schema da tabela de mensagens no DB — especificamente:
   - Nome do campo de conteúdo (`content`, `text`, `body`?)
   - Campos `role` e `conversation_id` existem?
   - Mensagens de `[[add_step:X]]`, steps internos e metadados são armazenados junto com o conteúdo ou em campo separado?

**Risco de schema:** se o campo de conteúdo armazenar markup interno (`[[add_step:X]]`, JSON de metadados, delimitadores do StepsPanel), o LLM vai receber ruído no histórico. Isso precisa ser confirmado antes — o fix pode exigir um campo `content_clean` ou uma função de strip antes de montar o payload.

---

### Escopo da mudança

#### Backend

**1. Configuração**

Adicionar ao `.env` / configuração:
```
CONTEXT_WINDOW_MESSAGES=20
```
- Tipo: `int`, default `20`
- Representa o número de mensagens (não pares), ordenadas por `created_at DESC LIMIT N`, depois invertidas para `ASC` antes de enviar.
- Não expor na UI nesta fase.

**Valor fixo: 10.** Valor único, sem distinção por provider. Documentar o tradeoff no `.env.example`.

**2. Query de histórico**

Antes de chamar o LLM, buscar as últimas N mensagens da conversa ativa:

```python
def load_conversation_history(conversation_id: str, limit: int) -> list[dict]:
    messages = db.query(
        "SELECT role, content FROM messages "
        "WHERE conversation_id = :conv_id "
        "  AND role IN ('user', 'assistant') "
        "ORDER BY created_at DESC LIMIT :limit",
        conv_id=conversation_id,
        limit=limit
    )
    messages.reverse()  # cronológico ASC
    return messages
```

Filtros obrigatórios:
- `role IN ('user', 'assistant')` — excluir `system`, `tool`, `tool_result`, metadados internos
- Se o campo `content` contiver markup interno: aplicar `strip_internal_markup(content)` antes de retornar

**3. Montagem do payload**

O histórico deve ser inserido *antes* da mensagem corrente, *depois* do system prompt:

```python
payload_messages = [
    {"role": "system", "content": system_prompt},
    *history,
    {"role": "user",  "content": current_message},
]
```

Para providers com formato diferente (Gemini `parts`, etc.), o adapter de provider já deve lidar com a conversão.

**4. Não fazer nesta fase**

- Token counting / truncagem por tokens: escopo do Chronicle.
- Compressão ou sumarização do histórico: escopo do Chronicle.
- Histórico de tool calls / steps para o LLM: escopo do Chronicle.

---

### Acceptance criteria

- [ ] Enviar 3+ mensagens em sequência; a Sabrina referencia contexto de mensagens anteriores na mesma conversa
- [ ] Iniciar nova conversa; contexto da conversa anterior não vaza
- [ ] Trocar de provider no meio de uma conversa não quebra o formato do histórico nem causa erro 400/422
- [ ] Conversa com 20+ mensagens: as últimas 20 são enviadas; não ocorre erro de payload excessivo
- [ ] Markup interno (`[[add_step:X]]`) não aparece no payload enviado ao LLM

---

### Riscos conhecidos

| Risco | Probabilidade | Mitigação |
|---|---|---|
| Campo `content` contém markup interno | Média | Strip antes de montar o payload; log do payload em modo debug |
| Adapter de provider rejeita role `assistant` (ex.: Gemini usa `model`) | Média | Confirmar mapeamento no adapter antes; adicionar normalização se necessário |
| N=20 estoura context window de modelos locais pequenos | Alta | Reduzir default para 10; doc no `.env.example` |
| Performance: query a cada mensagem sem índice | Baixa | Confirmar que `(conversation_id, created_at)` tem índice no DB |

## SPEC B-07 — Conversas não são renomeadas

**Prioridade:** Média
**Tipo:** Bug / UX
**Impacto:** Após 3-4 conversas, a lista lateral torna-se inutilizável — todos os títulos são idênticos.

---

### Decisão de design

**Opção A (MVP) — Determinístico:** Primeiros 60 chars da primeira mensagem do user.
**Opção B — LLM-gerado:** Call separado ao LLM ativo para gerar título semântico.

**Recomendação:** Opção A para B-07. Opção B pode ser adicionada como enhancement posterior.

---

### Escopo da mudança (Opção A)

#### Backend

**1. Função de geração de título**

```python
def generate_conversation_title(first_user_message: str) -> str:
    clean = " ".join(first_user_message.strip().split())
    if len(clean) <= 60:
        return clean
    return clean[:60].rstrip() + "…"
```

**2. Trigger**

- Ao persistir a **primeira resposta do assistant** em uma conversa.
- Condição: `conversation.title == DEFAULT_TITLE` (não sobrescrever renomeação manual futura).

**3. Persistência**

```sql
UPDATE conversations
SET title = :new_title
WHERE id = :conversation_id
  AND title = :default_title
```

A cláusula `AND title = :default_title` é a guard contra sobrescrever títulos já customizados.

**4. Notificação ao frontend**

Emitir via WebSocket após o UPDATE:
```json
{
  "type": "conversation_renamed",
  "conversation_id": "uuid-aqui",
  "title": "Título gerado"
}
```

#### Frontend

- Adicionar handler para `conversation_renamed` no listener de WebSocket.
- Atualizar o título na lista lateral.
- Se a conversa ativa for a renomeada, atualizar também o header/tab.

---

### Acceptance criteria

- [ ] Após primeira resposta do assistant, o título na sidebar reflete os primeiros 60 chars da primeira mensagem do user
- [ ] Mensagem curta (<60 chars) não adiciona "…"
- [ ] Refresh da página mantém o título correto (persistido no DB)
- [ ] Segunda mensagem em diante não sobrescreve o título já definido
- [ ] Múltiplas conversas simultâneas não se confundem

## SPEC B-08 — Layout de mensagens do agente: bolha → largura total

**Prioridade:** Baixa (visual)
**Tipo:** Bug / Produto
**Impacto:** Percepção de produto como chatbot genérico em vez de interface de agente profissional.

---

### Decisão de design

**Decisão:** Somente `role === 'assistant'` perde a bolha. Mensagens do user mantêm bolha alinhada à direita.

---

### Escopo da mudança

#### Frontend

**1. Identificar o componente de mensagem**

Localizar o componente responsável pela renderização.

**2. Mudança de estilo para `role === 'assistant'`**

Remover:
```css
background-color: var(--bubble-bg);
border-radius: 12px;
max-width: 80%;
padding: 8px 12px;
```

Adicionar:
```css
width: 100%;
background: transparent;
padding: 0;
```

**3. Garantias de renderização**

- Code blocks: o estilo é responsabilidade do markdown renderer, não da bolha.
- StepsPanel: verificar se herda `max-width` do container de bolha. Ajustar para `width: 100%`.
- Container de chat deve ter `max-width` razoável (ex.: 760px ou padding lateral).

---

### Acceptance criteria

- [ ] Mensagens do assistant ocupam 100% da largura do container de chat
- [ ] Nenhum background de bolha em mensagens do assistant
- [ ] Code blocks dentro de mensagens continuam com estilo próprio
- [ ] StepsPanel não perde largura
- [ ] Mensagens do user mantêm estilo anterior (bolha à direita)
- [ ] Responsividade: layout correto em 320px, 768px, 1280px
